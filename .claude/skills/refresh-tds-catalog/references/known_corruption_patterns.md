# Known DB corruption patterns — root cause + prevention

Documents the corruption patterns observed across multi-session DB
development, the pipeline steps that introduced them, and the safeguards
to prevent recurrence on future skill runs.

## Pattern A — Cross-material leakage via manufacturer-pool match

**Symptom**: A vendor entry's `tds_link` points to vendor A's TDS but the
mechanical values are vendor B's. Sometimes accompanied by `manufacturer`
field showing vendor B while `vendor_key` prefix says vendor A.

**Concrete cases caught**:
- Monel K-500 Nikon entry held EOS M400-4 heat-treated values
- Stainless Steel 17-4PH M300-4 held Inconel 718 M300-4 values
- AlSi10Mg M-290 60μm held Monel K-500 EOS values
- 316L 3D Systems entries briefly held 17-4PH values (fixed mid-session)

**Root cause**: An early version of `04_apply.py` pooled extracted records
by manufacturer and ran `best_match` across the entire manufacturer pool.
When the pool had records from multiple TDS files (different materials),
matches with the same (machine, layer, post-treatment) but wrong material
were chosen.

**Mitigation in current pipeline**:
- `04_apply.py` keys records by `tds_link` URL. A record from URL X can
  only match a DB entry whose `tds_link` is also X.
- `05_seed_new.py` writes the URL into the placeholder entry at creation
  time so the URL is fixed before any apply runs.

**Detection going forward** (`10_verify.py`):
- Phase 1 cross-checks vendor_key prefix vs `manufacturer` field vs URL
  host. Any disagreement flags the entry.
- Phase 3 fingerprints mechanical-value tuples and reports identical
  tuples across distinct materials.

## Pattern B — Within-material record cloning

**Symptom**: One DB material has many vendor entries pointing to the same
URL but only a few unique mechanical-value tuples — the same record was
applied to multiple distinct (machine, layer, post_treatment) keys.

**Concrete cases caught**:
- Nikon Ti6Al4V mds5137 — 15 entries / 6 unique tuples
- Nikon IN625 mds5127 — pre-fix had 13 entries / 1 tuple
- Nikon IN718 mds5128 — pre-fix had 8 entries / 1 tuple
- 3D Systems AlSi10Mg letter — 5 entries / 2 tuples
- EOS K-500 v=9 — 4 entries / 2 tuples
- Nikon Aheadd CP1 mds5144 — 4 entries / 2 tuples

**Root cause (two contributors)**:
1. **Lenient matcher**: `best_match` in early apply returned a record as
   long as ANY of (machine, layer, post) matched. If a TDS yielded one
   record but the DB had ten entries pointing at it, all ten received the
   one record's values.
2. **Sparse session-1 parsers**: The session-1 Nikon regex parser silently
   produced 0 records for some PDFs (mds5129, mds5147, mds5144), but
   parent placeholders had already been seeded. A second apply round
   filled them with a single fallback record cloned across all entries.

**Mitigation in current pipeline**:
- Strict matching: agent records carry explicit `machine`, `param_set`,
  `layer_thickness_um`, `post_treatment`; the apply step requires all
  three to match before overwriting.
- `_apply_strict.py` is the canonical re-applier: delete every DB entry
  pointing to URL X, then insert exactly the agent-extracted records for
  X. No cloning, no fuzzy reassignment.

**Detection going forward** (`10_verify.py`):
- Phase 3 lists duplicate fingerprints. The `_url_records.py` companion
  shows entries-per-URL vs unique-tuples-per-URL ratio — any ratio
  significantly > 1 indicates this pattern.

## Pattern C — Label drift

**Symptom**: `vendor_key` prefix, `manufacturer` field, and `machine`
field disagree even though `tds_link` is internally consistent.

**Concrete cases**:
- H13 "EOS [M290] @ ..." with `manufacturer: Nikon SLM Solutions`
- 316L "3D Systems [ProX DMP 320] @ ..." with `machine: ProX DMP 200`
  (the TDS itself is for DMP 200; key was generated by a later script
  that used the more recent model name)

**Root cause**: An early apply step overwrote the `manufacturer` and
`machine` fields when applying a record, but didn't rename the
`vendor_key` to match. Subsequent runs read the key as ground truth and
the field as data, producing inconsistency.

**Mitigation**: `_apply_strict.py` builds the vendor_key from the record
fields. Phase 1 of the verifier catches mismatches.

## Pattern D — Placeholder URLs that 404 silently

**Symptom**: Vendor entry has a TDS link that returns navigation HTML or
404, plus mechanical values that look plausible. Values came from cloning
during apply, not from the URL.

**Concrete cases**:
- `*.farsoon.com/material_datasheet_<X>_en.pdf` — fabricated URL pattern
  that never existed on Farsoon's site
- `*.huake3d.com/material_datasheet_<X>_en.pdf` — same
- `*.renishaw.com/en/material-datasheet-<X>--17862` — returns a
  navigation page with no mechanical data table
- `nikon-slm-solutions.com/materials/<X>/` — marketing pages, no MDS PDF

**Root cause**: A bulk seed step planted URLs in placeholders based on
naive URL guessing. The first apply round filled the placeholders with
the nearest-by-manufacturer record from another material's PDF.

**Mitigation**:
- `02_download.py` writes `error` to the manifest for any URL returning
  non-2xx or known-fake content.
- `09_drop_unverified.py` deletes any DB entry whose URL is in the
  manifest's error set. Prunes materials whose vendor list becomes empty.
- `assets/catalogs.json` is now a hand-maintained list — discovery picks
  up new TDS URLs but only from real catalog pages.

## Pattern E — Absurd hardness values

**Symptom**: HV field equals values like 5, 4287, 6507 (real HV for Ti is
~330-380; for IN738 ~350-450; for Ni-superalloy 280-500).

**Root cause**: Agent extraction occasionally mistakes column numbers for
hardness when the PDF table is rendered as positioned glyphs and column
boundaries collapse during pdftotext. Two specific failure modes:
- Reading an "elongation × 100" or "Rz" value as HV
- Combining two adjacent column values into one number

**Mitigation**: `11_autofix.py` nulls any HV outside [50, 800]. The
verifier (Phase 4) flags any value outside category-specific bands.

## Pattern F — Stale URLs after TDS revision

**Symptom**: `tds_link` returns 404 because the vendor re-published the
PDF with a new revision letter or date in the filename.

**Mitigation**: Skill is designed to re-discover URLs from
`assets/catalogs.json` every run. Never trust cached `tds_link` values;
the catalog is the source of truth.

## How the verifier catches each pattern

| Pattern | Verifier phase | Output field |
|---|---|---|
| A — cross-material leak | Phase 1, Phase 3 | label/link mismatch + cross-material tuple |
| B — within-material cloning | Phase 3 (Class 3 — same URL, multiple machines) | duplicate tuple |
| C — label drift | Phase 1 | key vs field |
| D — placeholder 404 | (manifest scan in apply) | _tds_unverified flag, then 09_drop |
| E — absurd HV | Phase 4 | out-of-range |
| F — stale URL | (discovery diff vs DB) | new-URLs-not-in-DB |

## Standard run order (post-fix)

```
01_discover  →  02_download  →  03_extract  →  10_verify  (sanity check)
            →  04_apply       (strict, URL-keyed)
            →  06_prune       (drop aggregator placeholders)
            →  09_drop_unverified  (nuke 404/fake-URL entries)
            →  10_verify  (final sanity check)
```

`11_autofix.py` and `_apply_strict.py` are intervention tools — only
needed when verify shows residual issues. Run them on demand.
